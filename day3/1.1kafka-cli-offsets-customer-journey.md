# Kafka CLI — Topics, Offsets & The Customer Journey (Easy Guide)

**What you'll learn in this demo, in order:**

1. Create a topic (with full explanation)
2. Send a customer journey: **CART → PAYMENT → ORDER** (with keys)
3. Read messages from the **beginning**
4. **Describe** the topic and **get the offsets**
5. Look inside **ONE partition** only
6. Jump to a **specific offset** ("give me message number 2")
7. Find **only ONE customer's** messages
8. See the consumer group's **bookmark** (current offset + lag)

**Setup used:** container `kafka-demo` (your separate CLI-demo Kafka — never clashes with the Spring Boot `kafka` container).

---

## STEP 0 — Make sure Kafka is running

```bash
docker ps
```

**Expected:** a row with `kafka-demo ... Up`.

If it's not there, start it:

```bash
docker run -d --name kafka-demo -p 9094:9092 apache/kafka:3.7.0
```

(If "name already in use": `docker start kafka-demo`)

---

## STEP 1 — Create the topic (with explanation)

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-topics.sh --create --topic customer-events --partitions 3 --replication-factor 1 --bootstrap-server localhost:9092
```

**Expected output:**

```
Created topic customer-events.
```

**Explanation, piece by piece:**

| Part | Meaning (simple words) |
|------|------------------------|
| `docker exec -it kafka-demo` | run the command INSIDE the Kafka container (the CLI tools live there, in `/opt/kafka/bin`) |
| `kafka-topics.sh` | the tool for everything about topics (create, list, describe, delete) |
| `--create` | the action: make a new topic |
| `--topic customer-events` | the topic's name — a named stream for ONE kind of data (like a WhatsApp group for customer events) |
| `--partitions 3` | split the topic into 3 ordered logs ("3 notebooks") → up to 3 consumers can work in parallel later |
| `--replication-factor 1` | keep 1 copy of the data — the max possible with 1 broker (production uses 3 brokers → 3 copies) |
| `--bootstrap-server localhost:9092` | the broker's address — every Kafka command needs this "first contact" address |

## STEP 2 — Send the customer journey (CART → PAYMENT → ORDER)

Start a **key-aware producer** (key = the customerId):

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-producer.sh --topic customer-events --bootstrap-server localhost:9092 --property parse.key=true --property key.separator=:
```

**Expected:** a `>` prompt appears.

Now type these 8 lines (ENTER after each) — two customers, deliberately mixed, like real traffic:

```
customer-101:CART added shoes
customer-102:CART added watch
customer-101:PAYMENT paid 2000rs
customer-102:PAYMENT paid 5000rs
customer-101:ORDER confirmed
customer-102:ORDER confirmed
customer-101:DELIVERED enjoy
customer-102:DELIVERED enjoy
```

Press `Ctrl+C` when done.

**Explanation:**

- `parse.key=true` → "my lines contain a key"
- `key.separator=:` → text before `:` = KEY (customerId), after = VALUE (the event)
- **The key decides the partition:** `partition = hash(key) % 3`. Same customer → same partition → their journey stays in perfect order (CART before PAYMENT before ORDER — always!)

## STEP 3 — Read everything from the BEGINNING (X-ray mode)

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic customer-events --from-beginning --bootstrap-server localhost:9092 --property print.key=true --property print.partition=true --property print.offset=true
```

**Expected output** (your partition numbers may differ — the PATTERN is what matters):

```
customer-101	Partition:0	Offset:0	CART added shoes
customer-102	Partition:2	Offset:0	CART added watch
customer-101	Partition:0	Offset:1	PAYMENT paid 2000rs
customer-102	Partition:2	Offset:1	PAYMENT paid 5000rs
customer-101	Partition:0	Offset:2	ORDER confirmed
customer-102	Partition:2	Offset:2	ORDER confirmed
customer-101	Partition:0	Offset:3	DELIVERED enjoy
customer-102	Partition:2	Offset:3	DELIVERED enjoy
```

Press `Ctrl+C` to stop (it says `Processed a total of 8 messages`).

**Explanation:**

- `--from-beginning` → "start from offset 0, give me the full history" (without it: only NEW messages from now on)
- `print.key / print.partition / print.offset` → X-ray glasses: see key + partition + offset of every message
- **READ the output:** customer-101 is ALWAYS in the same partition, offsets counting 0,1,2,3 — their journey in perfect order. Same for customer-102 in their own partition.

> **Note your two partition numbers here** (e.g. 101 → P0, 102 → P2). You'll need them in Steps 5–7.

## STEP 4 — Describe the topic + GET the offsets

### 4a. Describe = the topic's ID card

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-topics.sh --describe --topic customer-events --bootstrap-server localhost:9092
```

**Expected output:**

```
Topic: customer-events	TopicId: ...	PartitionCount: 3	ReplicationFactor: 1
	Topic: customer-events	Partition: 0	Leader: 1	Replicas: 1	Isr: 1
	Topic: customer-events	Partition: 1	Leader: 1	Replicas: 1	Isr: 1
	Topic: customer-events	Partition: 2	Leader: 1	Replicas: 1	Isr: 1
```

**Explanation:** 3 partitions exist; `Leader: 1` = broker 1 serves them; `Isr` = in-sync copies. Describe shows STRUCTURE — but not how many messages are inside. For that:

### 4b. Get the offsets = how many messages are in each partition

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-get-offsets.sh --topic customer-events --bootstrap-server localhost:9092
```

**Expected output** (format is `topic:partition:next-offset`):

```
customer-events:0:4
customer-events:1:0
customer-events:2:4
```

**Explanation — how to read it:**

| Line | Meaning |
|------|---------|
| `customer-events:0:4` | partition 0 has messages at offsets 0,1,2,3 → **4 messages** (the number shown = the NEXT offset to be written) |
| `customer-events:1:0` | partition 1 is **empty** (no customer hashed into it) |
| `customer-events:2:4` | partition 2 has 4 messages (the other customer's journey) |

> 4 + 0 + 4 = our 8 messages. This is how you check "where did my messages go?" without reading them.

## STEP 5 — Look inside ONE partition only

One partition = one customer's notebook. Read just that notebook (use YOUR partition number for customer-101 from Step 3):

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic customer-events --partition 0 --offset earliest --bootstrap-server localhost:9092 --property print.offset=true
```

**Expected output — ONLY that partition's messages, in perfect order:**

```
Offset:0	CART added shoes
Offset:1	PAYMENT paid 2000rs
Offset:2	ORDER confirmed
Offset:3	DELIVERED enjoy
```

`Ctrl+C` to stop.

**Explanation:**

- `--partition 0` → "open notebook 0 only"
- `--offset earliest` → "start from page 0" (when you use `--partition`, you say `--offset earliest` instead of `--from-beginning`)
- The full story reads top to bottom: cart → payment → order → delivered. **This is exactly what a payment-processing consumer would see.**

## STEP 6 — Jump to a SPECIFIC offset ("give me message number 2")

Start reading from offset 2 of that partition:

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic customer-events --partition 0 --offset 2 --bootstrap-server localhost:9092 --property print.offset=true
```

**Expected output — skips offsets 0 and 1, starts at 2:**

```
Offset:2	ORDER confirmed
Offset:3	DELIVERED enjoy
```

Want exactly ONE message? Add `--max-messages 1`:

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic customer-events --partition 0 --offset 2 --max-messages 1 --bootstrap-server localhost:9092 --property print.offset=true
```

**Expected output (prints one message and exits by itself):**

```
Offset:2	ORDER confirmed
Processed a total of 1 messages
```

**Explanation:**

- `--offset 2` = "put the bookmark on page 2 and read from there"
- `--max-messages 1` = "read 1 message and stop"
- Together: **the address of one exact message = partition + offset** ("notebook 0, page 2"). This is also how support teams replay one problematic message in production.

## STEP 7 — Find ONLY one customer's messages

The console consumer can't filter by key by itself — but we can filter the output. Print the key, and let the terminal filter the lines:

**Windows (CMD or PowerShell):**

```bash
docker exec kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic customer-events --from-beginning --bootstrap-server localhost:9092 --property print.key=true --timeout-ms 5000 | findstr customer-101
```

**Mac/Linux:** same command, replace `findstr customer-101` with `grep customer-101`.

**Expected output — only customer-101's lines:**

```
customer-101	CART added shoes
customer-101	PAYMENT paid 2000rs
customer-101	ORDER confirmed
customer-101	DELIVERED enjoy
```

**Explanation:**

- `print.key=true` → each line starts with the key, so we can filter on it
- `| findstr customer-101` → keep only lines containing "customer-101"
- `--timeout-ms 5000` → exit automatically after 5 seconds of no new messages (so the pipe finishes cleanly)
- Note: **no `-it`** after `docker exec` here — when piping output to another command, drop the `-it`

> **Teaching point:** in real projects you rarely filter like this — instead, the KEY already routed customer-101 to one partition, so the consumer that owns that partition naturally processes all their events. The filter trick is for debugging/support: "show me everything that happened to THIS customer."

## STEP 8 — The consumer group's BOOKMARK (current offset + lag)

So far our consumers were anonymous. Now let's read with a GROUP NAME, so Kafka saves a bookmark for us:

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic customer-events --from-beginning --group shop-group --bootstrap-server localhost:9092
```

Wait until all 8 messages print, then `Ctrl+C`.

Now ask Kafka: "where is shop-group's bookmark?"

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-consumer-groups.sh --describe --group shop-group --bootstrap-server localhost:9092
```

**Expected output:**

```
GROUP       TOPIC            PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
shop-group  customer-events  0          4               4               0
shop-group  customer-events  1          0               0               0
shop-group  customer-events  2          4               4               0
```

**Explanation — the three numbers that matter:**

| Column | Meaning |
|--------|---------|
| `CURRENT-OFFSET` | the group's bookmark: "I have read up to here" |
| `LOG-END-OFFSET` | the newest message in the partition (same numbers as Step 4b!) |
| `LAG` | END minus CURRENT = **how many messages behind** this group is |

**LAG = 0 everywhere** → the group is fully caught up. 🎉

**Prove the bookmark works:** send 2 more messages (Step 2 producer, e.g. `customer-101:REVIEW 5 stars`), then describe the group again → `LAG` becomes non-zero. Start the group consumer again (WITHOUT `--from-beginning`) → it reads ONLY the 2 new messages, from its bookmark. LAG back to 0.

> **This LAG number is the #1 Kafka health metric in production** — "how far behind are my consumers?"

## CHEAT SHEET — all commands in one table

| I want to... | Command (inside `docker exec -it kafka-demo /opt/kafka/bin/`) |
|---------------|------------------------------------------------|
| Create a topic | `kafka-topics.sh --create --topic T --partitions 3 --replication-factor 1 --bootstrap-server localhost:9092` |
| List topics | `kafka-topics.sh --list --bootstrap-server localhost:9092` |
| Describe a topic | `kafka-topics.sh --describe --topic T --bootstrap-server localhost:9092` |
| Count messages per partition | `kafka-get-offsets.sh --topic T --bootstrap-server localhost:9092` |
| Produce with keys | `kafka-console-producer.sh --topic T --bootstrap-server localhost:9092 --property parse.key=true --property key.separator=:` |
| Read all, from start, with X-ray | `kafka-console-consumer.sh --topic T --from-beginning --bootstrap-server localhost:9092 --property print.key=true --property print.partition=true --property print.offset=true` |
| Read ONE partition | `kafka-console-consumer.sh --topic T --partition 0 --offset earliest --bootstrap-server localhost:9092` |
| Jump to specific offset | `kafka-console-consumer.sh --topic T --partition 0 --offset 2 --bootstrap-server localhost:9092` |
| Read exactly N messages | add `--max-messages N` |
| Filter one customer | add `--property print.key=true --timeout-ms 5000` and pipe: `\| findstr customer-101` (drop `-it`) |
| See group bookmark + LAG | `kafka-consumer-groups.sh --describe --group G --bootstrap-server localhost:9092` |

---

## 30-SECOND RECAP

> - A **topic** is a set of ordered notebooks (**partitions**); the **key** decides which notebook — one customer, one notebook, perfect order (CART → PAYMENT → ORDER).
> - Every message's address = **partition + offset** ("notebook 0, page 2") — you can jump straight to it.
> - `kafka-get-offsets.sh` tells you **how full** each notebook is; `kafka-consumer-groups.sh --describe` tells you **where each reader's bookmark is** — and the gap between the two is **LAG**, the number production teams watch all day.
