# Kafka Hands-On CLI Demo — Session Guide

**Run this in the terminal BEFORE starting Spring Boot** — students see real messages travel with zero code.

| | |
|---|---|
| Container name | `kafka-demo` (UNIQUE — never clashes with your Spring Boot `kafka` container) |
| PC port | `9094` (your Spring Boot Kafka uses 9092 — both can run together!) |
| Broker address in commands | `localhost:9092` (we run commands INSIDE the container, so we use its internal port) |
| Topics used | `orders` (basics), `shop-events` (customer journey) |
| You need | Docker Desktop + 2 terminal windows side by side |

**Demo flow:** Setup → Create topic → Produce/Consume live → Magic moment (messages survive a dead consumer) → X-ray partitions & offsets → **The Customer Journey (keys + partitions, the grand finale)**

---

## PHASE 1 — SETUP

### Step 1 — Start Docker Desktop

Start Docker Desktop, wait until the whale icon is steady.

> **Say in demo:** Docker runs Kafka in a container — no Java/Kafka install needed on the PC. A container = a lightweight, disposable mini-computer.

### Step 2 — Start Kafka (separate demo container, no port clash!)

```bash
docker run -d --name kafka-demo -p 9094:9092 apache/kafka:3.7.0
```

**Expected output:**

```
3f9a1c2b4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9e0f1a
```

(one long random string = the container ID)

> **Say in demo:**
> - `-d` = run in background
> - `--name kafka-demo` = a UNIQUE name — completely separate from the `kafka` container used in the Spring Boot demos
> - `-p 9094:9092` = PC port **9094** → container port 9092. We chose 9094 on purpose: your Spring Boot Kafka already owns 9092, so this one NEVER clashes with it. Both can run at the same time.
> - Image = official Apache Kafka in **KRaft mode** (self-coordinating — no ZooKeeper needed)
>
> **Note for you:** every command in this demo runs INSIDE the container (`docker exec`), where Kafka listens on its own port 9092 — that's why the commands say `localhost:9092` even though the PC port is 9094. Nothing to change, it just works.

## PHASE 2 — VERIFY

### Step 3 — Is it running?

```bash
docker ps
```

**Expected output:**

```
CONTAINER ID   IMAGE                COMMAND        STATUS         PORTS                    NAMES
3f9a1c2b4d5e   apache/kafka:3.7.0   "/etc/kafka…"  Up 2 minutes   0.0.0.0:9094->9092/tcp   kafka-demo
```

(If your Spring Boot `kafka` container is also running, you'll see TWO rows — that's fine, they don't clash.)

> **Tip:** `docker ps` shows RUNNING containers only; `docker ps -a` shows stopped ones too. Error *"name already in use"*? An old stopped `kafka-demo` exists — remove it with `docker rm -f kafka-demo` and run Step 2 again.

### Step 4 — Is Kafka itself alive?

```bash
docker logs kafka-demo --tail 5
```

**Expected output (the important line):**

```
... INFO Kafka Server started (kafka.server.KafkaRaftServer)
```

> **Say in demo:** This reads Kafka's own startup log. This line = the broker is alive and listening on port 9092.

## PHASE 3 — TOPICS

### Step 5 — List topics (empty, fresh Kafka)

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-topics.sh --list --bootstrap-server localhost:9092
```

**Expected output:**

```
(empty — no output, no error)
```

> **Say in demo:**
> - `docker exec -it kafka-demo ...` = run a tool INSIDE the container (Kafka's CLI tools live in `/opt/kafka/bin`)
> - `--bootstrap-server` = the broker address every Kafka client must be given as first contact
> - Empty list = fresh Kafka, no topics yet

### Step 6 — Create the topic

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-topics.sh --create --topic orders --partitions 3 --replication-factor 1 --bootstrap-server localhost:9092
```

**Expected output:**

```
Created topic orders.
```

> **Say in demo:**
> - **TOPIC** = a named stream (like a WhatsApp group for one kind of data)
> - `--partitions 3` = split the topic into 3 ordered logs → enables 3 parallel consumers later
> - `--replication-factor 1` = one copy of the data (max possible with 1 broker; production uses 3)

### Step 7 — Describe it

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-topics.sh --describe --topic orders --bootstrap-server localhost:9092
```

**Expected output:**

```
Topic: orders	TopicId: ...	PartitionCount: 3	ReplicationFactor: 1
	Topic: orders	Partition: 0	Leader: 1	Replicas: 1	Isr: 1
	Topic: orders	Partition: 1	Leader: 1	Replicas: 1	Isr: 1
	Topic: orders	Partition: 2	Leader: 1	Replicas: 1	Isr: 1
```

> **Say in demo:** 3 independent partitions exist. `Leader: 1` = broker id 1 serves all reads/writes. `Isr` = In-Sync Replicas = which copies are up to date (only broker 1 here).

## PHASE 4 — PRODUCE & CONSUME (2 terminals side by side!)

### Step 8 — TERMINAL 1 (LEFT): start a consumer

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic orders --bootstrap-server localhost:9092
```

**Expected output:**

```
(cursor blinks, prints nothing — this is CORRECT)
```

> **Say in demo:** Silence is correct — it is connected and PULLING ("anything new?") in a loop. Kafka consumers always pull; the broker never pushes. Default start position = "from now onwards" (skips old messages).

### Step 9 — TERMINAL 2 (RIGHT): start a producer

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-producer.sh --topic orders --bootstrap-server localhost:9092
```

**Expected output:**

```
>
```

> **Say in demo:** Each line typed becomes one RECORD (message) sent to topic `orders`. The producer only knows the BROKER — it has no idea who (or if anyone) consumes. That is **DECOUPLING**.

### Step 10 — Type messages in TERMINAL 2 (ENTER after each)

```
order-1 shoes
order-2 tshirt
order-3 watch
```

**Expected in TERMINAL 1 (within milliseconds each):**

```
order-1 shoes
order-2 tshirt
order-3 watch
```

> **Say in demo:** Per message: broker receives it → APPENDS it to the end of one partition's log file ON DISK → stamps it with an OFFSET (position number). The consumer's next poll fetches & prints it. ~2 ms total.

## PHASE 5 — THE MAGIC MOMENT 🎩

### Step 11 — Kill the consumer, keep producing

1. **TERMINAL 1:** press `Ctrl+C` (consumer is now DEAD 💀)
2. **TERMINAL 2:** type 2 more messages:

```
order-4 mobile
order-5 laptop
```

**Expected:** producer shows **NO error** — it works fine with zero consumers alive.

> **Ask the class:** *"Is order-4 lost?"* — then run Step 12.
> Messages go to the broker's DISK, not to the consumer. That's why nothing broke.

### Step 12 — Resurrect the consumer with full history

**TERMINAL 1:**

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic orders --from-beginning --bootstrap-server localhost:9092
```

**Expected output — ALL 5 messages, including the 2 sent while the consumer was dead** (order may look shuffled):

```
order-2 tshirt
order-1 shoes
order-3 watch
order-4 mobile
order-5 laptop
```

> **KEY LESSON to say out loud:** Kafka STORED the messages; offline consumers lose nothing; reading NEVER deletes. **Kafka = diary, not to-do list.**
> Shuffled order = messages were spread over 3 partitions — the next step makes this visible.

## PHASE 6 — PARTITIONS & OFFSETS MADE VISIBLE (X-ray mode)

### Step 13 — Show partition + offset per message

**TERMINAL 1:** `Ctrl+C`, then:

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic orders --from-beginning --bootstrap-server localhost:9092 --property print.partition=true --property print.offset=true
```

**Expected output (your numbers will vary):**

```
Partition:2	Offset:0	order-1 shoes
Partition:0	Offset:0	order-2 tshirt
Partition:2	Offset:1	order-3 watch
Partition:1	Offset:0	order-4 mobile
Partition:0	Offset:1	order-5 laptop
```

> **Say in demo — now you can SEE:**
> - (a) messages spread over partitions 0, 1, 2 (no key → Kafka balances them)
> - (b) each partition has its OWN offset counter starting at 0
> - (c) within one partition the order is PERFECT — but there is NO global order
>
> **Plant the problem:** *"Imagine these were the steps of ONE customer's shopping: cart, payment, delivery — scattered over 3 partitions, read by 3 different consumers... payment could be processed BEFORE add-to-cart! How do we fix this?"* → Phase 7.

## PHASE 7 — THE CUSTOMER JOURNEY 🛒 (keys + partitions, the grand finale)

**The story to tell:** In a real shop, ONE customer's events MUST stay in order:

```
ADDED_TO_CART  →  PAYMENT_DONE  →  ORDER_SHIPPED  →  DELIVERED
```

If `PAYMENT_DONE` is processed before `ADDED_TO_CART`, the system breaks.
The fix: **key = customerId** → all events of one customer go to ONE partition → perfect order for that customer, while different customers still spread across partitions (parallelism).

### Step 14 — Create a fresh topic for the journey

**TERMINAL 2:** `Ctrl+C` first, then:

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-topics.sh --create --topic shop-events --partitions 3 --replication-factor 1 --bootstrap-server localhost:9092
```

**Expected output:**

```
Created topic shop-events.
```

> **Say in demo:** Fresh topic, 3 partitions again — but this time we will send messages WITH A KEY.

### Step 15 — TERMINAL 1: start the X-ray consumer (shows KEY + partition + offset)

**TERMINAL 1:** `Ctrl+C` the old consumer if still running, then:

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-consumer.sh --topic shop-events --from-beginning --bootstrap-server localhost:9092 --property print.key=true --property print.partition=true --property print.offset=true
```

**Expected output:**

```
(cursor blinks, waiting — correct)
```

> **Say in demo:** `print.key=true` — this time we also SEE the key of every message. Full X-ray: key + partition + offset + value.

### Step 16 — TERMINAL 2: start the KEY-aware producer

```bash
docker exec -it kafka-demo /opt/kafka/bin/kafka-console-producer.sh --topic shop-events --bootstrap-server localhost:9092 --property parse.key=true --property key.separator=:
```

**Expected output:**

```
>
```

> **Say in demo:**
> - `parse.key=true` — "my typed lines contain a key"
> - `key.separator=:` — text before `:` = KEY (the customerId), after = VALUE (the event)
> - Kafka decides the partition from the key: `partition = hash(key) % 3`

### Step 18 — READ the output with the class (the most important 2 minutes)

Point at the output and extract the three lessons:

**Lesson 1 — same key = same partition, ALWAYS:**

```
customer-101  ->  Partition 0, every single time
customer-102  ->  Partition 2, every single time
customer-103  ->  Partition 1, every single time
```

**Lesson 2 — read ONE partition top to bottom = ONE customer's perfect story:**

```
Partition 0 (customer-101's notebook):
   Offset 0 : ADDED_TO_CART shoes
   Offset 1 : PAYMENT_DONE 2000rs
   Offset 2 : ORDER_SHIPPED
   Offset 3 : DELIVERED enjoy          <- cart -> pay -> ship -> deliver. PERFECT ORDER.

Partition 2 (customer-102's notebook):
   Offset 0 : ADDED_TO_CART watch
   Offset 1 : PAYMENT_DONE 5000rs
   Offset 2 : ORDER_SHIPPED
   Offset 3 : DELIVERED enjoy          <- also perfect, independently!

Partition 1 (customer-103's notebook):
   Offset 0 : ADDED_TO_CART book
   Offset 1 : PAYMENT_FAILED retry-later   <- failed journey, still in order
```

**Lesson 3 — parallelism is kept:** 3 consumers (one per partition) could process 3 customers AT THE SAME TIME, and each would still see its customer's events in perfect order.

> **Say to the class:** *"The key gives you order WHERE YOU NEED IT (per customer), and parallelism EVERYWHERE ELSE (across customers). That's the whole trick. In real projects the key is customerId, orderId, driverId, deviceId — whatever thing must keep its own history in order."*

### Step 19 — Optional proof: replay one customer's journey only

You can read ONE partition (one customer's notebook) by itself:

```bash

```

**Expected output — ONLY customer-101's journey, in exact order (use the partition number YOUR output showed for customer-101):**

```
Offset:0	ADDED_TO_CART shoes
Offset:1	PAYMENT_DONE 2000rs
Offset:2	ORDER_SHIPPED
Offset:3	DELIVERED enjoy
```

> **Say in demo:** `--partition 0 --offset earliest` = "give me notebook 0, from page 0". One partition = one clean, ordered story. This is exactly what a payment-processing consumer would see.

# PARTITIONS & OFFSETS — THE WHITEBOARD SCRIPT

## 1. The analogy (draw this)

A TOPIC is not one notebook. It is a **SET of notebooks**. Topic `shop-events` with 3 partitions = 3 notebooks, and thanks to keys, **each customer writes in their own notebook**:

```
Notebook P0 = customer-101 :  CART | PAYMENT | SHIPPED | DELIVERED
Notebook P1 = customer-103 :  CART | PAYMENT_FAILED
Notebook P2 = customer-102 :  CART | PAYMENT | SHIPPED | DELIVERED
                              ^page0  ^page1    ^page2    ^page3   <- offsets
```

- **PARTITION** = one notebook (an ordered, append-only log)
- **OFFSET** = the page number INSIDE that notebook
- **KEY** = decides WHICH notebook (same key → always the same notebook)
- **Writing** = always on the NEXT EMPTY PAGE (append-only: never insert in the middle, never erase a page)

## 2. Why does Kafka split a topic into partitions?

ONE notebook = only one person can write/read it efficiently. THREE notebooks = three consumers can each take one and work IN PARALLEL — and the notebooks can even live on three different servers (brokers).

**Partitions = Kafka's unit of scaling.** More partitions → more parallel consumers possible.
(Rule: max useful consumers in one group = number of partitions.)

## 3. Which partition does a message go to?

**CASE A — no key** (Phase 4-6 of the demo):
Kafka spreads messages across partitions for balance (round-robin/sticky).
GOOD: even load. BAD: related messages get separated → a customer's PAYMENT could be processed before their ADD_TO_CART. 💥

**CASE B — with a key** (Phase 7, `customer-101:...`):

```
partition = hash(key) % number_of_partitions

hash("customer-101") % 3  ->  always 0   (for example)
hash("customer-102") % 3  ->  always 2
hash("customer-103") % 3  ->  always 1
```

Same key ALWAYS lands in the same partition.
GOOD: all events of customer-101 stay in one notebook, in perfect order (cart → payment → shipped → delivered). Other customers hash to other notebooks → parallelism is kept.

## 4. The ordering rule (memorize — top interview question)

> **Kafka guarantees order ONLY WITHIN a partition, NEVER across partitions.**

- Need order per customer? → key = customerId (what we demoed)
- Need TOTAL order of everything? → 1 partition only (no parallelism — almost never the right answer)

## 5. Offsets — three jobs of one number

| Job | Meaning |
|-----|---------|
| **ADDRESS** | "partition 0, offset 1" identifies exactly one message — "customer-101's PAYMENT_DONE" in our demo |
| **BOOKMARK** | each consumer group remembers "I have read up to offset X in partition Y" — stored BY KAFKA in an internal topic (`__consumer_offsets`); consumer restarts → continues from its bookmark |
| **REPLAY** | messages stay on disk, so moving the bookmark backwards re-reads history (`--from-beginning`, or `--partition 0 --offset earliest` like Step 19) |

**Important details:**

- Offsets start at 0 and only ever increase
- Each partition has its OWN independent counter — P0 has offsets 0,1,2... and P2 also has 0,1,2... They are different messages!
- Offsets are never reused, even after old messages expire (retention deletes pages but page numbers keep counting up)
- **LAG** = newest offset in partition − consumer's bookmark = "how many pages behind is the reader" — the #1 Kafka health metric in production

## 6. The 15-second summary for the class

> *"A topic is a set of ordered notebooks (partitions). The KEY decides which notebook a message goes into — one customer, one notebook. Every message lands on the next page, and the page number is its offset. Kafka never rips out a page when someone reads it — every reader just keeps their own bookmark."*
