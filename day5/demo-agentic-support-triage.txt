=======================================================================
    DEMO : AGENTIC AI SUPPORT TRIAGE  (Apache Camel + Spring Boot)
    one simple real-world app — enterprise transformation in action
=======================================================================

THE REAL-WORLD STORY.

    Every company has a support inbox. Hundreds of emails a day:

        "hi i was charged twice for my order please help!!"
        "when will my package arrive?"
        "I want a refund, order O-2 never came"

    TODAY (before):  a HUMAN reads every email, looks up the
                     customer, checks the order, decides what to do.
                     Slow, expensive, boring.

    AFTER (this app): Camel watches the inbox. An AI AGENT reads
                     each email, CALLS OUR JAVA CODE to look up the
                     customer and the order, and DECIDES:

                         URGENT  -> escalate to a human now
                         REFUND  -> send to the refunds team
                         NORMAL  -> standard queue

                     Camel routes the ticket to the right folder
                     with the agent's analysis attached.

    That is "agentic AI for enterprise transformation" in one
    sentence: the reading, looking-up and deciding — the human
    part — is now done by an agent. Humans handle only outcomes.

ARCHITECTURE

    tickets/inbox/*.txt                (customer emails arrive here)
           |
           v
    +---------------------------------------------------+
    |  SPRING BOOT + CAMEL                              |
    |                                                   |
    |  Camel route: file -> agent -> decide -> route    |
    |                        |                          |
    |                        v                          |
    |            SupportAgent (Spring AI)               |
    |            reads the email, may call:             |
    |              findCustomer(email)   <- our Java    |
    |              getOrderStatus(id)    <- our Java    |
    |            answers: URGENT / REFUND / NORMAL      |
    |                        |                          |
    |                        v                          |
    |            OLLAMA in Docker (:11434)              |
    |            model qwen2.5:3b — free, local         |
    +------------------------|--------------------------+
                             |
            +----------------+----------------+
            v                v                v
     tickets/urgent/  tickets/refund/  tickets/normal/
     (email + the agent's analysis inside each file)

    CAMEL  = watches the inbox, routes the outcome  (integration)
    AGENT  = reads, investigates via tools, decides (intelligence)
    TOOLS  = plain Java methods with dummy data     (your systems)

=======================================================================
    STEP 3 : THE TOOLS  (the agent's hands — plain Java)
=======================================================================

FILE : src/main/java/com/gosi/supporttriage/ai/SupportTools.java
-----------------------------------------------------------------------

    package com.gosi.supporttriage.ai;

    import org.springframework.ai.tool.annotation.Tool;

    import java.util.Map;

    public class SupportTools {

        // dummy CRM — in real life: a REST call / database
        private static final Map<String, String> CUSTOMERS = Map.of(
                "rahul@mail.com", "Rahul, GOLD customer since 2019",
                "neha@mail.com",  "Neha, GOLD customer since 2021",
                "bob@mail.com",   "Bob, BASIC customer since 2024");

        // dummy order system
        private static final Map<String, String> ORDERS = Map.of(
                "O-1", "DELIVERED on time",
                "O-2", "PAYMENT FAILED, charged twice by mistake",
                "O-3", "SHIPPED, arriving in 2 days");

        @Tool(description = "Look up a customer by their email address. "
                + "Returns the customer's name and membership level, or NOT_FOUND.")
        public String findCustomer(String email) {
            System.out.println(">>> TOOL: findCustomer(" + email + ")");
            return CUSTOMERS.getOrDefault(email.toLowerCase().trim(), "NOT_FOUND");
        }

        @Tool(description = "Get the current status of an order by its order id, "
                + "for example O-1 or O-2. Returns the status or NOT_FOUND.")
        public String getOrderStatus(String orderId) {
            System.out.println(">>> TOOL: getOrderStatus(" + orderId + ")");
            return ORDERS.getOrDefault(orderId.toUpperCase().trim(), "NOT_FOUND");
        }
    }

-----------------------------------------------------------------------

NOTE
    The @Tool DESCRIPTION is written FOR THE MODEL — it reads it to
    decide when to call which method. The println lines are your
    window: every agent investigation is visible in the console.


=======================================================================
    STEP 4 : THE AGENT  (reads, investigates, decides)
=======================================================================

FILE : src/main/java/com/gosi/supporttriage/ai/SupportAgent.java
-----------------------------------------------------------------------

    package com.gosi.supporttriage.ai;

    import org.springframework.ai.chat.client.ChatClient;
    import org.springframework.stereotype.Service;

    @Service
    public class SupportAgent {

        private final ChatClient chatClient;

        public SupportAgent(ChatClient.Builder builder) {
            this.chatClient = builder
                    .defaultSystem("""
                        You are a support triage agent for an online shop.
                        You receive one customer email at a time.

                        Investigate with the tools:
                        - find the customer by their email address
                        - check any order id mentioned (like O-1, O-2)

                        Then decide. Your answer MUST start with exactly
                        one of these words on the first line:
                          URGENT  - money problems (double charge, failed
                                    payment) or angry GOLD customers
                          REFUND  - customer clearly asks for a refund
                          NORMAL  - everything else

                        After the decision word, write a 2-sentence
                        summary of the case for the human team.
                        Never invent customer or order data - if the
                        tools return NOT_FOUND, say so.
                        """)
                    .build();
        }

        /** Camel calls this: email text in -> "DECISION + summary" out */
        public String triage(String emailText) {
            return chatClient.prompt()
                    .tools(new SupportTools())     // the agent's toolbox
                    .user(emailText)
                    .call()
                    .content();
        }
    }

-----------------------------------------------------------------------

NOTE
    The system prompt IS the job description: what to investigate,
    what decisions exist, output format pinned down hard. Strict
    "first word = decision" makes the answer machine-routable —
    that is how AI output plugs into a Camel choice().


=======================================================================
    STEP 5 : THE CAMEL ROUTE  (watch -> triage -> route)
=======================================================================

FILE : src/main/java/com/gosi/supporttriage/routes/TriageRoute.java
-----------------------------------------------------------------------

    package com.gosi.supporttriage.routes;

    import com.gosi.supporttriage.ai.SupportAgent;
    import org.apache.camel.builder.RouteBuilder;
    import org.springframework.stereotype.Component;

    @Component
    public class TriageRoute extends RouteBuilder {

        @Override
        public void configure() throws Exception {

            from("file:tickets/inbox")
                .routeId("support-triage")
                .convertBodyTo(String.class)
                .log("NEW TICKET ${file:name}:\n${body}")

                // keep the original email — the agent call replaces the body
                .setProperty("email", body())

                // the agent reads, investigates (tools!), decides
                .bean(SupportAgent.class, "triage")
                .log("AGENT VERDICT:\n${body}")

                // first word of the verdict -> routing header
                .process(e -> {
                    String verdict = e.getIn().getBody(String.class).trim();
                    String decision = verdict.split("\\s+")[0]
                            .replaceAll("[^A-Z]", "");   // URGENT / REFUND / NORMAL
                    e.getIn().setHeader("decision", decision);

                    // final file content = original email + agent analysis
                    String email = e.getProperty("email", String.class);
                    e.getIn().setBody("=== CUSTOMER EMAIL ===\n" + email
                            + "\n\n=== AGENT ANALYSIS ===\n" + verdict);
                })

                // route the ticket where it belongs
                .choice()
                    .when(header("decision").isEqualTo("URGENT"))
                        .log("-> tickets/urgent  (a human is needed NOW)")
                        .to("file:tickets/urgent")
                    .when(header("decision").isEqualTo("REFUND"))
                        .log("-> tickets/refund")
                        .to("file:tickets/refund")
                    .otherwise()
                        .log("-> tickets/normal")
                        .to("file:tickets/normal")
                .end();
        }
    }

-----------------------------------------------------------------------

NOTE — the division of labor, one line each:
    Camel  : WHEN (file arrives) and WHERE (which folder)
    Agent  : WHAT IS THIS and WHAT SHOULD HAPPEN
    choice : trusts only the FIRST WORD — and has an otherwise(),
             because AI output is input: never trust, always route
             the unexpected somewhere visible.

=======================================================================
    THE ENTERPRISE TRANSFORMATION SLIDE  (say this at the end)
=======================================================================

    BEFORE                            AFTER (this app)
    ------------------------------    ------------------------------
    human reads every email           agent reads every email
    human searches CRM + orders       agent calls the tools itself
    human decides the queue           agent decides, Camel routes
    minutes per ticket                seconds per ticket
    humans do ALL tickets             humans do only URGENT ones,
                                      with the analysis pre-written

    And notice what we did NOT throw away: the file component, the
    stash-in-property idiom, choice/otherwise, "every message ends
    somewhere visible" — the integration discipline carries the AI.
    The agent is just the smartest step a Camel route ever had.


=======================================================================
    IF SOMETHING GOES WRONG
=======================================================================

    Connection refused :11434        Ollama container not running
    "model not found"                qwen2.5:3b not pulled (Step 0)
    Verdict without URGENT/REFUND/   small-model wobble -> otherwise()
    NORMAL first word                catches it -> tickets/normal
                                     (by design! discuss it)
    Agent answers but no >>> TOOL    asked nothing tool-worthy, or
    lines                            model skipped tools -> make the
                                     email mention the order id
    Very slow (30-60 s)              normal: agent = several model
                                     calls in a loop, on CPU
    Route does nothing               @Component missing, or file
                                     dropped in the wrong folder —
                                     tickets/ is relative to the
                                     project root
